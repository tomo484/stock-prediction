## 次に作る製品を作成するために必要な要素を書き連ねる

## 絶対に守るべき基準（必須）

### バックエンド
- **3層アーキテクチャの厳守**
  - Repository層、Service層、Controller層を明確に分離する
  - 各層の責務を守る（Repository: DB操作、Service: ビジネスロジック、Controller: HTTP処理）
  - 層を跨ぐ依存関係は一方向のみ（Controller → Service → Repository）

- **CRUD APIの実装**
  - 最低限1つのリソースに対して完全なCRUD操作を実装する
  - RESTfulなエンドポイント設計を守る

- **型安全性の確保**
  - DTO（Data Transfer Object）を使用してリクエスト/レスポンスの型を定義する
  - Goの型システムを活用する

- **エラーハンドリング**
  - 適切なHTTPステータスコードを返す
  - エラーメッセージを統一する

### フロントエンド
- **React Query（TanStack Query）の使用**
  - データフェッチングは必ずReact Queryを使用する
  - カスタムフック（useQuery, useMutation）を作成する
  - キャッシュ戦略を意識する

- **型安全性の確保**
  - TypeScriptを必ず使用する
  - バックエンドの型定義と一致させる
  - `any`型の使用は最小限にする（可能な限り避ける）

- **コンポーネント設計**
  - 再利用可能なコンポーネントを作成する
  - 各ページコンポーネントは`"use client"`を明示する

## 絶対にしてはいけない基準（禁止事項）

### バックエンド
- **アーキテクチャの混在を禁止**
  - Controller層で直接DB操作を行わない
  - Service層でHTTP処理を行わない
  - Repository層でビジネスロジックを含めない

- **認証の実装を禁止**（今回は認証なしで進める）
  - JWT認証は実装しない
  - 認証ミドルウェアは使用しない
  - ユーザー認証関連のエンドポイントは作成しない

- **複雑な機能の実装を禁止**
  - 画像アップロード機能は実装しない（今回は除外）
  - ページネーションは実装しない（シンプルな一覧表示のみ）
  - 検索・フィルタリング機能は実装しない

### フロントエンド
- **状態管理ライブラリの混在を禁止**
  - Redux、Zustand、Recoilなどの状態管理ライブラリは使用しない
  - React Query以外のデータフェッチングライブラリ（SWRなど）は使用しない
  - グローバル状態管理はReact Queryのみで行う

- **クラスコンポーネントの使用を禁止**
  - 関数コンポーネントのみを使用する
  - Hooks以外の方法で状態管理を行わない

- **インラインスタイルの多用を禁止**
  - Tailwind CSSを使用する
  - インラインスタイル（style属性）は最小限にする

## 推奨事項（任意だが推奨）

### バックエンド
- エラーハンドリングの統一化（カスタムエラー型の定義）
- ロギングの実装
- 環境変数の適切な管理

### フロントエンド
- ローディング状態の適切な表示
- エラー状態の適切な表示
- 再利用可能なUIコンポーネントの作成

## 技術スタック（shopmarketと同様）

- Go

- GORM
- PostgreSQL
- Docker（開発環境）

### フロントエンド
- Next.js（App Router）
- TypeScript
- Tailwind CSS
- React Query（TanStack Query）
- React Hook Form
- Zod
- Axios

## プロジェクト規模の目安

- **リソース数**: 1〜2つのリソース（例: ブログ記事、タスク、メモなど）
- **ページ数**: 5〜7ページ程度
- **開発期間**: 1〜2週間を目安
- **機能の複雑さ**: shopmarketよりシンプル、でも完全なCRUD操作

## 学習目標

1. **アーキテクチャパターンの理解**
   - 3層アーキテクチャの実践的な理解
   - 責務の分離の重要性を体感する

2. **型安全性の徹底**
   - バックエンドとフロントエンドの型定義の整合性
   - TypeScriptの型システムの活用

3. **React Queryの習熟**
   - キャッシュ戦略の理解
   - カスタムフックの設計パターン

4. **コード品質の向上**
   - 再利用可能なコンポーネント設計
   - エラーハンドリングの統一
